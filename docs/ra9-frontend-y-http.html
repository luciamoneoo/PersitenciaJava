1. Arquitectura cliente-servidor

Cuando abres un navegador y escribes una URL, estás usando un modelo llamado cliente-servidor. El cliente (el navegador) hace una petición; el servidor la procesa y devuelve una respuesta. Entre ellos viajan mensajes siguiendo las reglas del protocolo HTTP.

El stack completo de CinemaApp

Nuestra aplicación tiene varias capas. Cada capa tiene una responsabilidad clara y se comunica solo con la que tiene justo encima y justo debajo:

🌐 Navegador (Chrome, Firefox…)
HTML + CSS + JavaScript · el cliente
↕ HTTP (peticiones y respuestas JSON)
⚙️ Spring Boot (Tomcat embebido)
@RestController · Lógica de negocio
↕ JPA / Hibernate
🗂️ Spring Data JPA
Repositorios · Entidades · JPQL
↕ JDBC (protocolo de MySQL)
🐬 MySQL en Docker
Base de datos relacional · ra9db

Cuando el usuario hace clic en "Películas" en nuestra interfaz, el navegador envía GET /peliculas a Spring Boot. Spring llama al servicio, que llama al repositorio, que ejecuta SELECT * FROM pelicula en MySQL. La respuesta sube la cadena hasta que el controlador la serializa como JSON y la devuelve al navegador, donde JavaScript la convierte en tarjetas HTML visibles.

Tomcat embebido: el servidor dentro del JAR

A diferencia de un servidor web clásico (Apache, Nginx…) que hay que instalar y configurar aparte, Spring Boot incluye Apache Tomcat dentro del propio JAR. Cuando ejecutas mvn spring-boot:run, arranca un servidor HTTP en el puerto 8080 que:

  • Atiende peticiones HTTP de cualquier cliente (navegador, curl, Postman…)
  • Enruta cada URL al @RestController correcto
  • Sirve los ficheros estáticos de src/main/resources/static/ directamente
El directorio src/main/resources/static/ es especial en Spring Boot: todo lo que pongas ahí se sirve como si fuera un servidor web estático. Nuestro index.html está en esa carpeta, por eso puedes abrirlo con http://localhost:8080/.

2. Protocolo HTTP

HTTP (HyperText Transfer Protocol) es el protocolo de comunicación de la web. Define exactamente cómo se formatean y envían los mensajes entre cliente y servidor. Toda petición tiene una estructura fija y toda respuesta también.

Estructura de un request (petición)

Cuando el navegador hace fetch('/peliculas'), envía al servidor un mensaje con este formato: 

GET /peliculas HTTP/1.1               ← línea de petición: método + URL + versión
Host: localhost:8080                   ← cabecera obligatoria: host destino
Accept: application/json               ← le decimos al servidor qué formato queremos
Content-Type: application/json         ← en POST/PUT: formato del cuerpo que enviamos
                                       ← línea en blanco que separa cabeceras del cuerpo
← (sin cuerpo en GET)

En un POST que crea una película, el mensaje incluye además un cuerpo JSON:

POST /peliculas HTTP/1.1
Host: localhost:8080
Content-Type: application/json
Content-Length: 97

{
  "titulo": "Oppenheimer",
  "anio": 2023,
  "duracion": 180,
  "precioEntrada": 9.50,
  "director": { "id": 1 }
}

Estructura de un response (respuesta)

La respuesta del servidor tiene un formato similar:

HTTP/1.1 200 OK                        ← línea de estado: versión + código + descripción
Content-Type: application/json;charset=UTF-8
Transfer-Encoding: chunked

[
  {
    "id": 1,
    "titulo": "Inception",
    "anio": 2010,
    "duracion": 148,
    "precioEntrada": 8.50,
    "nombreDirector": "Christopher Nolan",
    "generos": ["Ciencia ficción", "Thriller"]
  },
  ...
]

Métodos HTTP y su relación con CRUD

HTTP define varios métodos (también llamados verbos). En REST, cada verbo HTTP tiene una semántica clara relacionada con las operaciones CRUD:

Método HTTP Operación CRUD Endpoint en CinemaApp Código de respuesta esperado
GET Read (leer) GET /peliculas 200 OK
GET Read (leer uno) GET /peliculas/3 200 OK · 404 si no existe
POST Create (crear) POST /peliculas 201 Created
PUT Update (actualizar) PUT /peliculas/3 200 OK
DELETE Delete (eliminar) DELETE /peliculas/3 204 No Content
GET es idempotente y seguro: puedes repetirlo mil veces sin efecto secundario. POST no es idempotente: cada POST puede crear un nuevo recurso. DELETE es idempotente: borrar algo que ya no existe no cambia el estado del sistema.

Códigos de estado HTTP

El código de estado (un número de 3 dígitos) le dice al cliente si la petición tuvo éxito o qué tipo de error ocurrió. El primer dígito indica la categoría:

RangoCategoríaEjemplos habituales
2xx Éxito 200 OK – petición exitosa con cuerpo
201 Created – recurso creado (POST)
204 No Content – éxito sin cuerpo (DELETE)
4xx Error del cliente 400 Bad Request – datos inválidos (validación fallida)
404 Not Found – recurso no encontrado
409 Conflict – conflicto (ej. sin asientos disponibles)
5xx Error del servidor 500 Internal Server Error – excepción no controlada
503 Service Unavailable – servidor sobrecargado o caído

En Spring Boot, los 4xx los produce tu código (p.ej. al lanzar RecursoNoEncontradoException → 404). Los 5xx indican un bug en el servidor o un problema de infraestructura.

Cabeceras (Headers) importantes

CabeceraQuién la envíaSignificado
Content-Type: application/json Cliente (en POST/PUT) El cuerpo de la petición es JSON
Content-Type: application/json;charset=UTF-8 Servidor La respuesta contiene JSON codificado en UTF-8
Accept: application/json Cliente El cliente prefiere recibir JSON (Spring lo usa para content negotiation)
Authorization: Bearer <token> Cliente Token de autenticación (JWT, OAuth…) – no se usa en este proyecto

3. Frontend vs Backend

En el desarrollo web moderno hablamos de frontend y backend para referirnos a las dos grandes mitades de una aplicación:

AspectoFrontendBackend
¿Dónde se ejecuta? En el navegador del usuario En el servidor (nuestra máquina o la nube)
¿Qué hace? Muestra la interfaz, gestiona eventos del usuario, llama a la API Lógica de negocio, acceso a la base de datos, seguridad
Tecnologías HTML, CSS, JavaScript (en este proyecto: vanilla JS) Java, Spring Boot, JPA, MySQL (en este proyecto)
Ficheros clave index.html, estilos.css, api.js, app.js PeliculaController.java, PeliculaService.java, PeliculaRepository.java
¿Cómo se comunican? JSON sobre HTTP — el frontend hace peticiones, el backend responde

Separación de responsabilidades

Una de las ideas fundamentales del desarrollo web es que el frontend y el backend no deben mezclar sus responsabilidades. El backend no debería generar HTML; el frontend no debería acceder directamente a la base de datos. Entre ellos solo pasa JSON.

Esta separación tiene ventajas prácticas:

  • El mismo backend puede servir a un frontend web, una app móvil y una aplicación de escritorio.
  • Los equipos de frontend y backend pueden trabajar en paralelo.
  • Es más fácil reemplazar una parte sin tocar la otra.

Nuestro caso: un solo servidor, mismo origen

En CinemaApp, Spring Boot hace de backend y también de servidor de ficheros estáticos. El frontend (HTML + JS) y la API REST conviven en http://localhost:8080. Esto es posible porque Spring Boot sirve automáticamente la carpeta src/main/resources/static/.

En proyectos más grandes, el frontend suele ser una aplicación separada con su propio servidor (p. ej. React con npm run dev en el puerto 3000). En ese caso, el frontend y el backend están en orígenes distintos y necesitas configurar CORS (lo veremos en la sección 6).

4. APIs REST y JSON

¿Qué es una API REST?

REST (Representational State Transfer) es un estilo de arquitectura para APIs web. Una API REST sigue estos principios:

  • Recursos como URLs: cada entidad del dominio tiene su propia URL. /peliculas es la colección; /peliculas/3 es un recurso concreto.
  • Verbos HTTP: la operación la indica el método HTTP (GET, POST, PUT, DELETE), no la URL. Nunca verás /getPelicula o /deletePelicula en una API REST.
  • Sin estado (stateless): cada petición debe contener toda la información necesaria para procesarla. El servidor no recuerda las peticiones anteriores.
  • Representación: el servidor puede devolver el recurso en varios formatos. Hoy en día se usa casi siempre JSON.

JSON – JavaScript Object Notation

JSON es el formato de intercambio de datos más usado en APIs REST. Su sintaxis es muy sencilla: objetos entre llaves {}, arrays entre corchetes [], y valores separados por comas. 

// Así luce un PeliculaDTO serializado como JSON
{
  "id": 1,
  "titulo": "Inception",
  "anio": 2010,
  "duracion": 148,
  "sinopsis": "Un ladrón entra en los sueños de otros...",
  "precioEntrada": 8.50,
  "nombreDirector": "Christopher Nolan",
  "generos": ["Ciencia ficción", "Thriller"]
}

Reglas básicas de JSON:

  • Las claves van siempre entre comillas dobles: "titulo"
  • Los strings también: "Inception"
  • Los números sin comillas: 2010, 8.50
  • Los booleanos: true / false
  • El valor nulo: null
  • No se permiten comentarios en JSON real (los de arriba son solo didácticos)

5. La API fetch() de JavaScript

fetch() es la función nativa del navegador para hacer peticiones HTTP desde JavaScript. Está disponible en todos los navegadores modernos sin instalar nada. Es el corazón de nuestro fichero api.js.

async / await — código asíncrono legible

Las peticiones de red no son inmediatas: el navegador tiene que enviar la petición, esperar a que viaje por la red y a que el servidor procese y responda. Durante ese tiempo, el resto de la página debe seguir funcionando (el usuario puede hacer clic en otros elementos, el ratón sigue moviéndose…).

JavaScript resuelve esto con el concepto de asincronía. Una función async puede pausarse en un punto (await) sin bloquear el resto del programa, y reanudarse cuando llega el resultado. 

// Una función async devuelve implícitamente una Promesa.
// await pausa la función hasta que la Promesa se resuelve.
// El navegador no se bloquea — solo esta función queda "pausada".

async function ejemplo() {
  console.log('Antes de la petición');

  const resultado = await fetch('/peliculas'); // ← pausa aquí

  console.log('La petición ha terminado');
  // resultado es un objeto Response, no los datos todavía
}

GET con fetch() – leer datos del servidor

Este es el código real de getPeliculas() en nuestro api.js, explicado línea a línea: 

// Equivalente HTTP:  GET http://localhost:8080/peliculas
async function getPeliculas() {

  // 1. Llamamos a fetch() con la URL. Sin segundo argumento, usa GET por defecto.
  const respuesta = await fetch('/peliculas');

  // 2. respuesta.ok es true si el código HTTP está entre 200 y 299.
  if (!respuesta.ok) {
    throw new Error(`Error HTTP ${respuesta.status}`);
  }

  // 3. .json() lee el cuerpo del response y lo parsea.
  // También es asíncrono → necesita otro await.
  const datos = await respuesta.json();

  // 4. Devolvemos el array de PeliculaDTO ya como objeto JavaScript.
  return datos;
}
fetch() no lanza error en códigos 4xx/5xx. Solo lanza error si hay un fallo de red (sin conexión, timeout…). Por eso siempre debemos comprobar respuesta.ok manualmente.

POST con fetch() – enviar datos al servidor

Para crear un recurso necesitamos enviar datos en el cuerpo de la petición, indicar el formato con el header Content-Type y cambiar el método a POST: 

// Equivalente HTTP:  POST /peliculas   (con JSON en el cuerpo)
async function crearPelicula(datos) {

  const respuesta = await fetch('/peliculas', {

    method: 'POST',           // cambiamos el método HTTP

    headers: {
      'Content-Type': 'application/json',  // le decimos al servidor que enviamos JSON
    },

    // JSON.stringify() convierte el objeto JavaScript a string JSON.
    // Sin esto el servidor recibiría "[object Object]", no JSON.
    body: JSON.stringify(datos),
  });

  if (!respuesta.ok) throw new Error(`Error HTTP ${respuesta.status}`);

  // Spring devuelve 201 Created con el recurso creado en el cuerpo
  return respuesta.json();
}

DELETE con fetch() – eliminar un recurso

DELETE es especial: si tiene éxito, el servidor devuelve 204 No Content, que significa "lo borré pero no tengo nada que devolverte". Intentar llamar a .json() en una respuesta 204 daría error. 

// Equivalente HTTP:  DELETE /peliculas/3
async function eliminarPelicula(id) {

  const respuesta = await fetch(`/peliculas/${id}`, {
    method: 'DELETE',
  });

  if (!respuesta.ok) throw new Error(`Error HTTP ${respuesta.status}`);

  // 204 No Content → no hay cuerpo que parsear, devolvemos null
  if (respuesta.status === 204) return null;

  return respuesta.json();
}

Manejo de errores con try/catch

Cuando usamos await, los errores se pueden capturar con un bloque try/catch exactamente igual que con código síncrono: 

async function cargarPeliculas() {
  try {

    // Todo lo que puede fallar va dentro del try
    const peliculas = await getPeliculas();
    renderizarPeliculas(peliculas);

  } catch (err) {

    // Si fetch() lanzó un error (red) o nosotros hicimos throw,
    // llegamos aquí. err.message contiene el texto del error.
    console.error('Fallo al cargar películas:', err.message);
    mostrarMensaje(err.message, 'error');  // barra de estado roja

  }
}

Actualizar el DOM con los datos recibidos

Una vez que tenemos los datos del servidor, los mostramos en pantalla manipulando el DOM (la representación en memoria del HTML). La forma más directa es con innerHTML y template literals (las comillas invertidas ` `): 

function renderizarPeliculas(peliculas) {

  // Array.map() transforma cada película en un string HTML.
  // Los template literals permiten incrustar expresiones con ${}.
  const html = peliculas.map(p => `
    <div class="card-item">
      <div class="card-titulo">${p.titulo}</div>
      <div class="card-meta">${p.anio} · ${p.duracion} min</div>
      <div class="precio-tag">${p.precioEntrada} €</div>
    </div>`).join('');  // unimos todos los strings en uno solo

  // Asignamos el HTML resultante al contenedor del DOM.
  // El navegador lo parsea y muestra las tarjetas.
  document.getElementById('lista-peliculas').innerHTML = html;
}
Usar innerHTML con datos que vienen de usuarios externos puede ser un riesgo de seguridad (XSS – Cross-Site Scripting). En este proyecto los datos vienen de nuestra propia API controlada, así que es seguro. En producción con datos de usuarios desconocidos deberías sanear el texto con textContent o una biblioteca de sanitización.

6. CORS y @CrossOrigin

La política del mismo origen (Same-Origin Policy)

Los navegadores tienen una regla de seguridad llamada Same-Origin Policy: por defecto, una página web solo puede hacer peticiones fetch() al mismo origen desde el que fue cargada. El origen está compuesto por:

  • Esquema: http o https
  • Host: nombre del servidor o IP
  • Puerto: número de puerto

Dos URLs tienen el mismo origen solo si los tres coinciden:

URL del frontendURL del backend¿Mismo origen?
http://localhost:8080http://localhost:8080/peliculas✅ Sí (nuestro caso)
http://localhost:3000http://localhost:8080/peliculas❌ No (puerto distinto)
https://miapp.comhttp://miapp.com/peliculas❌ No (esquema distinto)
http://miapp.comhttp://api.miapp.com/peliculas❌ No (host distinto)

¿Qué es CORS?

CORS (Cross-Origin Resource Sharing) es el mecanismo que permite al servidor autorizar peticiones de orígenes cruzados. Funciona mediante cabeceras HTTP especiales que el servidor añade a sus respuestas.

Cuando el navegador va a hacer una petición a un origen distinto con un método "no simple" (PUT, DELETE, o POST con Content-Type: application/json), primero envía automáticamente una petición preflight con el método OPTIONS para preguntar al servidor si lo permite: 

// Petición preflight automática del navegador (tú no la escribes)
OPTIONS /peliculas HTTP/1.1
Host: localhost:8080
Origin: http://localhost:3000
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type

// Respuesta del servidor (con CORS habilitado)
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type

Nuestro caso: ¿por qué NO necesitamos CORS?

En CinemaApp, el fichero index.html es servido por el propio Tomcat de Spring Boot en http://localhost:8080. Cuando el JavaScript hace fetch('/peliculas'), la petición va al mismo servidor (http://localhost:8080). El origen es idéntico → no hay petición cross-origin → no hace falta CORS.

Si abres las DevTools (F12 → Network) en CinemaApp y filtras por "Fetch/XHR", verás que no aparece ninguna petición OPTIONS. Esto confirma que el navegador reconoce que estamos en el mismo origen y no necesita hacer el preflight.

¿Cuándo sí haría falta CORS?

Si en el futuro migras el frontend a React o Vue y lo ejecutas con npm run dev (que usa el puerto 3000 por defecto), tendrías dos servidores: el frontend en http://localhost:3000 y el backend en http://localhost:8080. Orígenes distintos → necesitas CORS.

En Spring Boot se habilita con la anotación @CrossOrigin:

// En el controlador, para permitir peticiones desde localhost:3000
@CrossOrigin(origins = "http://localhost:3000")
@RestController
@RequestMapping("/peliculas")
public class PeliculaController { ... }

// O globalmente en una clase de configuración:
@Configuration
public class WebConfig implements WebMvcConfigurer {

  @Override
  public void addCorsMappings(CorsRegistry registry) {
    registry.addMapping("/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("GET", "POST", "PUT", "DELETE");
  }
}
